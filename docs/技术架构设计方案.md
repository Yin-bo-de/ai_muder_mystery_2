# AI案件解谜应用 技术架构设计方案

## 1. 技术架构总览

### 1.1 整体架构分层
```
┌─────────────────────────────────────────────────────────────┐
│                        前端层 (React)                        │
│  页面组件 / 状态管理(Zustand) / API封装 / 工具函数            │
└───────────────────────────────────┬─────────────────────────┘
                                    │ HTTP/JSON
┌───────────────────────────────────▼─────────────────────────┐
│                        后端层 (FastAPI)                      │
│  API路由层 / 业务服务层 / Agent层 / 数据模型层 / 工具层       │
└───────────────────────────────────┬─────────────────────────┘
                                    │ SDK调用
┌───────────────────────────────────▼─────────────────────────┐
│                        AI模型层                              │
│  OpenAI GPT系列 / Anthropic Claude系列 / 其他兼容大模型       │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心技术栈选型
| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 后端语言 | Python 3.10+ | 生态完善，LangChain原生支持，开发效率高 |
| Web框架 | FastAPI | 高性能，自动生成API文档，类型注解支持好 |
| AI框架 | LangChain | 成熟的Agent开发框架，支持LCEL链式调用，方便管理Agent上下文 |
| 日志框架 | loguru | 结构化日志，配置简单，支持文件轮转和多字段绑定 |
| 前端框架 | React 18 + Vite | 开发体验好，生态完善，构建速度快 |
| 状态管理 | Zustand | 轻量级，API简洁，比Redux更易用，适合中小型项目 |
| UI组件库 | Ant Design | 组件丰富，样式统一，开箱即用，减少UI开发成本 |
| 测试框架 | pytest (后端) / Jest (前端) | 生态完善，文档丰富 |

### 2.2 核心模块设计

#### 2.2.1 案件生成Agent (case_generator_agent.py)
**职责：** 生成完整、逻辑自洽的案件数据
**核心能力：**
- 生成死者信息（身份、死因、死亡时间、社会关系）
- 生成3-5个嫌疑人信息（背景、动机、时间线、与死者关系、性格人设）
- 设计作案手法、真凶身份、完整证据链
- 合理分布线索（场景物证、人物证词、关联线索、解密线索）
- 生成3个干扰性剧情分支种子
**输入：** 案件题材、难度级别（v2.0支持）
**输出：** `Case` 数据模型对象

#### 2.2.2 嫌疑人Agent (suspect_agent.py)
**职责：** 每个嫌疑人角色独立实例，处理对话逻辑，保证人设一致性
**核心设计：**
- 每个Agent拥有独立的上下文记忆，严格控制上下文窗口大小
- 固定的人设prompt模板，包含角色背景、已知信息、隐藏秘密、行为规则
- 支持三种对话模式：单独审讯、全体质询、被其他角色质询
- 动态压力值系统：根据提问内容调整角色状态（镇定/紧张/愤怒/心虚）
- 真凶专属逻辑：自动说谎、掩盖证据、嫁祸他人，无明显逻辑漏洞
**输入：** 用户提问、上下文消息、当前场景状态
**输出：** 角色回复内容、状态更新

#### 2.2.3 对话管理器 (dialogue_manager.py)
**职责：** 控制对话流程、发言优先级、频率，保证游戏体验
**核心规则：**
- 三级优先级架构：
  - P0：用户直接提问/指令（必须立即响应）
  - P1：被点名/诬陷/出示不利证据/单独审讯
  - P2：主动发言（压力释放、剧情推进、自然互动）
- 发言频率控制：
  - 正常探案：<10%主动发言，每1-2分钟1次
  - 激烈对峙：<20%主动发言，最多30秒1次
  - 玩家卡关：100%提示性发言
- 插话机制：时间线冲突时最多2次插话/反驳
- 用户控场指令优先级最高，所有角色必须立即服从

#### 2.2.4 裁判Agent (judge_agent.py)
**职责：** 结案阶段处理、案件正确性判断、复盘报告生成
**核心能力：**
- 评估用户推理准确率（基于线索收集完整度、逻辑合理性）
- 判断指认是否正确，给出明确反馈
- 生成完整案件还原报告
- 生成用户探案过程复盘：推理亮点、遗漏点、准确率统计
- 控制错误指认次数（最多2次）

#### 2.2.5 矛盾检测引擎 (contradiction_detector.py)
**职责：** 分析对话历史，识别潜在矛盾点，生成系统提示
**核心能力：**
- 分析对话历史，匹配预定义的contradiction_points
- 支持三种矛盾类型：timeline（时间线）、spatial（空间/视野）、evidence（证据）
- 生成给用户的系统提示（如"张小姐的卧室窗户能看到车库，但她说没看到任何人？"）
- 判断是否满足AI角色之间的反驳触发条件
- 与案件生成器配合，自动生成对应的线索供用户勘查发现

#### 2.2.6 反驳决策引擎 (refusal_decision_engine.py)
**职责：** 评估威胁程度，决定是否进行反驳，避免无止境对话
**核心能力：**
- 威胁程度评估：分析被质疑的点对自己的威胁有多大
- 反驳能力评估：检查是否有足够的counter_evidence进行反驳
- 人设因素考虑：根据性格决定是否反驳（易怒的人更容易反驳）
- 反驳次数限制：每轮对话最多2次反驳
- 智能决策逻辑：
  - 威胁足够大 + 有证据 → 高概率反驳
  - 威胁大但无证据 → 低概率反驳（或转移话题）
  - 威胁小 → 不反驳
- 与嫌疑人Agent配合，生成自然的反驳回复

### 2.3 会话状态管理
**会话数据结构：**
```python
class ContradictionPoint:
    """矛盾点模型"""
    contradiction_id: str         # 矛盾点唯一标识
    type: str                     # 矛盾类型：timeline | spatial | evidence
    description: str              # 矛盾描述（如"张小姐的卧室窗户能看到车库"）
    involved_suspects: List[str]  # 涉及的嫌疑人ID列表
    trigger_condition: Dict[str, Any]  # 触发条件配置
    hint_for_user: str            # 给用户的系统提示文案
    related_clue_id: Optional[str] # 关联的线索ID（如果有的话）

class GameSession:
    session_id: str              # 会话唯一标识
    user_id: str                 # 用户ID
    case: Case                   # 完整案件数据
    game_status: GameStatus      # 游戏状态：进行中/已结案/已失败
    collected_clues: List[Clue]  # 已收集线索列表
    dialogue_history: List[Message] # 完整对话历史
    suspect_states: Dict[str, SuspectState] # 各嫌疑人状态
    user_operations: List[UserOperation] # 用户操作记录
    wrong_guess_count: int       # 错误指认次数
    start_time: datetime         # 开始时间
    last_active_time: datetime   # 最后活跃时间
    contradiction_points: List[ContradictionPoint] # 本案件的矛盾点列表
    refusal_count: int           # 本轮对话已发生的反驳次数
```
**存储方案：**
- 开发阶段：内存存储 + 定期持久化到JSON文件
- 生产阶段：Redis存储（过期时间24小时） + MySQL持久化游戏记录

### 3.2 状态管理设计
**gameStore (全局游戏状态)：**
- 会话ID、案件基本信息、游戏状态、游戏时间
- 嫌疑人列表及状态
- 已收集线索统计
- 当前模式（勘查/质询/结案）

**clueStore (线索状态)：**
- 所有已收集线索列表
- 线索分类（物证/证词/关联/解密）
- 当前场景线索列表
- 线索关联状态

**dialogueStore (对话状态)：**
- 对话历史消息列表
- 当前对话模式（全体/单独）
- 当前正在对话的嫌疑人
- 系统提示消息列表

### 3.3 核心交互流程
1. **现场勘查交互：**
   - 用户点击场景物品 → 调用勘查接口 → 获得线索 → 更新线索库
   - 解密线索：用户输入密码/选择关联线索 → 验证成功后解锁线索

2. **对话质询交互：**
   - 单独审讯：选择嫌疑人 → 进入私密对话空间（背景变暗，标记【保密】）→ 发送问题 → 获得回复
   - 全体质询：
     - 输入框输入"@"时弹出嫌疑人候选列表，支持多选
     - 发送消息（支持@指定角色）→ 按优先级顺序获得回复
     - AI角色消息以打字机效果逐字显示
     - 系统自动检测矛盾并以特殊样式提示
     - 符合条件时触发AI角色之间的反驳/插话（最多2次）

3. **指认凶手交互：**
   - 选择凶手 → 填写动机、作案手法、时间线 → 选择证据 → 提交 → 获得结果

### 3.4 核心前端增强功能
1. **@嫌疑人功能：**
   - 基于Ant Design Mentions组件或自定义实现
   - 输入"@"时触发，显示嫌疑人头像+姓名列表
   - 支持键盘导航选择
   - 支持多选@多个嫌疑人
   - 已选择的嫌疑人以标签形式显示，可删除

2. **打字机效果：**
   - React useState + setTimeout实现
   - 可配置速度（根据人设调整：紧张的人语速变化）
   - 支持跳过动画（用户点击可直接显示完整内容）
   - 增加沉浸感和真实感

3. **系统提示样式：**
   - 特殊背景色（如黄色/橙色渐变）突出显示
   - 前缀【系统提示】高亮
   - 可能的矛盾点以加粗/红色强调
   - 支持收起/展开长提示

4. **单独审讯增强：**
   - 背景变暗效果（半透明遮罩）
   - 对话气泡标记【保密】标签
   - 其他嫌疑人头像变灰，显示"隔离中"
   - 场景变为"审讯室"风格

5. **线索关联功能：**
   - 拖拽线索到关联区域
   - 或点击"关联线索"按钮多选
   - 满足条件时系统自动提示可解密
   - 关联成功后生成新线索

6. **卡关提示功能：**
   - 用户超过10分钟无进展时触发
   - 或连续3次无意义对话时触发
   - 提示形式：系统提示建议勘查某个场景/询问某个问题
   - 可选择"不需要提示"关闭

## 4. 核心API接口设计

### 4.1 游戏管理接口
| 接口 | 方法 | 路径 | 参数 | 返回值 |
|------|------|------|------|--------|
| 创建新案件 | POST | `/api/v1/game/new` | - | `session_id`, `case_basic_info` |
| 获取游戏状态 | GET | `/api/v1/game/{session_id}/status` | `session_id` | 完整游戏状态 |
| 提交勘查操作 | POST | `/api/v1/game/{session_id}/investigate` | `session_id`, `scene`, `item` | 新线索（如果找到） |
| 提交线索解密 | POST | `/api/v1/game/{session_id}/decrypt` | `session_id`, `clue_id`, `password`/`related_clues` | 解密结果 |
| 提交指认 | POST | `/api/v1/game/{session_id}/accuse` | `session_id`, `suspect_id`, `motive`, `modus_operandi`, `evidence` | 指认结果、结案报告 |

### 4.2 对话交互接口
| 接口 | 方法 | 路径 | 参数 | 返回值 |
|------|------|------|------|--------|
| 发送消息 | POST | `/api/v1/agent/{session_id}/send` | `session_id`, `message`, `target_suspects`（可选） | 角色回复列表、系统提示 |
| 切换审讯模式 | POST | `/api/v1/agent/{session_id}/mode` | `session_id`, `mode`（`single`/`group`）, `suspect_id`（可选） | 切换结果 |
| 执行控场指令 | POST | `/api/v1/agent/{session_id}/command` | `session_id`, `command` | 执行结果 |

## 5. 核心流程时序设计

### 5.1 开局流程
```
用户点击开始新案件
    ↓
后端调用案件生成Agent生成完整案件
    ↓
创建游戏会话，初始化状态
    ↓
返回案件基础信息、嫌疑人列表、初始场景给前端
    ↓
前端渲染游戏主界面，显示系统提示
```

### 5.2 线索搜查流程
```
用户点击场景物品，点击勘查按钮
    ↓
前端调用勘查接口
    ↓
后端clue_service判断是否找到线索
    ↓
如果找到线索，添加到已收集线索列表
    ↓
返回线索信息给前端
    ↓
前端更新场景线索列表和线索库
```

### 5.3 对话交互流程
```
用户发送消息（可@指定角色，支持@候选列表）
    ↓
前端调用发送消息接口
    ↓
后端dialogue_manager解析消息意图，确定回复角色
    ↓
调用对应嫌疑人Agent依次生成回复
    ↓
    ┌─────────────────────────────────┐
    │  矛盾检测引擎分析对话历史      │
    │  - 匹配预定义contradiction_points │
    │  - 识别时间线/空间/证据矛盾      │
    └─────────────────────────────────┘
    ↓
发现矛盾？
    ↓ 是
    ┌─────────────────────────────────┐
    │  生成用户系统提示（如"张小姐    │
    │  的卧室窗户能看到车库..."）      │
    └─────────────────────────────────┘
    ↓
    ┌─────────────────────────────────┐
    │  反驳决策引擎评估               │
    │  - 评估威胁程度                 │
    │  - 检查反驳证据                 │
    │  - 考虑人设因素                 │
    │  - 限制反驳次数（≤2次）         │
    └─────────────────────────────────┘
    ↓
满足反驳条件？
    ↓ 是
    ┌─────────────────────────────────┐
    │  插入反驳消息（最多2条）        │
    │  - 相关嫌疑人主动插话           │
    │  - 自然融入对话流程             │
    └─────────────────────────────────┘
    ↓
返回回复内容和系统提示给前端
    ↓
前端渲染消息气泡（打字机效果）
    ↓
展示系统提示（特殊样式高亮）
```

### 5.4 反驳决策流程
```
收到可能触发反驳的消息
    ↓
评估威胁程度
    ↓
    ┌─────────────────────────────────┐
    │  威胁评分（0-1）：              │
    │  • 被直接指证 → 高（0.8-1.0）  │
    │  • 被出示关键证据 → 高          │
    │  • 时间线被质疑 → 中（0.5-0.7）│
    │  • 一般性询问 → 低（<0.5）      │
    └─────────────────────────────────┘
    ↓
检查反驳能力
    ↓
    ┌─────────────────────────────────┐
    │  是否有counter_evidence？       │
    │  • 有确凿证据 → 可以反驳        │
    │  • 有部分理由 → 可能反驳        │
    │  • 无证据 → 不反驳/转移话题     │
    └─────────────────────────────────┘
    ↓
考虑人设因素
    ↓
    ┌─────────────────────────────────┐
    │  性格修正系数：                 │
    │  • 暴躁/冲动 → ×1.5            │
    │  • 冷静/谨慎 → ×0.7            │
    │  • 压力值高 → ×1.2              │
    └─────────────────────────────────┘
    ↓
最终决策：威胁 × 证据 × 人设 > 阈值？
    ↓ 是 + 本轮反驳次数 < 2
    ┌─────────────────────────────────┐
    │  生成反驳回复                   │
    │  • 符合人设语气                 │
    │  • 使用counter_evidence         │
    │  • 标记本轮反驳次数+1           │
    └─────────────────────────────────┘
```

## 6. 性能与可扩展性设计

### 6.1 性能优化
- **Agent上下文管理：** 每个嫌疑人Agent独立维护上下文，定期清理过时消息，控制上下文窗口在4k token以内
- **异步处理：** 大模型调用使用异步非阻塞方式，避免阻塞主线程
- **缓存机制：** 常用提示词模板、静态资源使用内存缓存，减少IO操作
- **消息压缩：** 对话历史使用摘要算法压缩，减少token消耗

### 6.2 可扩展性设计
- **多模型兼容：** 设计统一的模型调用接口，方便后续接入不同厂商的大模型
- **插件化设计：** Agent行为规则、线索类型、玩法机制使用插件化设计，方便后续扩展新功能
- **题材扩展：** 案件生成Agent支持不同题材模板（现代/古装/科幻/恐怖），v2.0可快速支持
- **多租户支持：** 会话管理设计支持多用户同时在线，无状态后端方便水平扩展

### 6.3 稳定性设计
- **降级机制：** 大模型调用失败时，使用预设回复模板，保证游戏不会中断
- **超时控制：** 所有模型调用设置超时时间（默认30秒），超时自动重试2次
- **错误处理：** 完善的异常捕获和错误提示，用户友好的错误信息
- **日志监控：** 所有关键节点（案件生成、模型调用、用户操作）都有完整日志，方便问题定位

## 7. 部署方案

### 7.1 开发环境
- 后端：`uvicorn main:app --reload --port 8000` 自动重载
- 前端：`npm run dev` 启动开发服务器，端口5173
- 不需要数据库，使用内存存储会话

### 7.2 生产环境
- 后端：Docker容器化部署，使用Gunicorn + Uvicorn
- 前端：构建后使用Nginx部署
- 缓存：Redis存储会话数据，过期时间24小时
- 数据库：MySQL存储用户游戏记录、案件库
- 负载均衡：Nginx反向代理，支持多实例水平扩展
- 监控：接入日志服务、性能监控、错误告警
